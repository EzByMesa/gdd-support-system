# 05. Описание функциональности

## Анонимные агенты

### Зачем

Пользователь не видит реальных имён сотрудников поддержки. Это защищает личные данные агентов и предотвращает возможность адресного давления на конкретного сотрудника.

### Как работает

Каждому агенту в каждом тикете присваивается уникальный псевдоним формата `"Прилагательное Животное"`. Псевдоним генерируется при первом сообщении агента в тикет и хранится в таблице `AgentAlias`.

### Словари

- **50 прилагательных:** Ласковый, Добрый, Смелый, Мудрый, Весёлый, Тёплый, Нежный, Яркий, Славный, Милый, Бодрый, Верный, Храбрый, Чуткий, Дружный, Светлый, Тихий, Быстрый, Ловкий, Пушистый, Сонный, Забавный, Уютный, Солнечный, Звёздный, Облачный, Снежный, Лунный, Радужный, Искристый, Шустрый, Озорной, Мечтательный, Задумчивый, Кроткий, Ясный, Вольный, Мирный, Цветочный, Ванильный, Карамельный, Бархатный, Сказочный, Волшебный, Хрустальный, Жемчужный, Изумрудный, Рубиновый, Золотой, Серебряный
- **50 животных:** Котёнок, Пингвин, Лисёнок, Зайчонок, Медвежонок, Совёнок, Ёжик, Бельчонок, Оленёнок, Дельфин, Панда, Коала, Хомячок, Кролик, Черепашка, Попугай, Утёнок, Воробушек, Филин, Барсучок, Тигрёнок, Львёнок, Жирафик, Слонёнок, Бегемотик, Выдра, Бобёр, Енотик, Хорёк, Лемур, Фламинго, Колибри, Снегирь, Ласточка, Журавлик, Тюлень, Моржонок, Китёнок, Капибара, Шиншилла, Альпака, Горностай, Песец, Сурок, Ленивец, Носорожик, Пони, Росомаха, Морской Конёк, Косатка
- **Итого комбинаций:** 50 × 50 = 2 500

### Уникальность в рамках тикета

При генерации система проверяет уже использованные псевдонимы в данном тикете. Если тикет обрабатывали несколько агентов (после делегирований), у каждого будет свой псевдоним. Повтор псевдонима в одном тикете исключён (до 100 попыток генерации).

### Видимость

| Кто смотрит | Что видит |
|------------|----------|
| USER | Только псевдоним ("Ласковый Котёнок"), `id: "agent"` |
| AGENT/ADMIN | Реальное имя + псевдоним в скобках ("Иванов И.И.", alias: "Ласковый Котёнок") |

Анонимизация применяется в:
- История сообщений (REST API `GET /api/tickets/:id/messages`)
- Новые сообщения (WebSocket `message`)
- Индикатор набора (WebSocket `typing`)
- Уведомления (`body` уведомления содержит псевдоним)
- После делегирования (`agent_changed` содержит новый псевдоним)

## Делегирование тикетов

### Назначение

Агент может передать тикет другому агенту. Принимающий агент должен подтвердить передачу. После принятия назначение обновляется, и пользователь получает уведомление о смене агента (с новым псевдонимом).

### Жизненный цикл

```
Агент A (assignee) → POST /api/tickets/:id/delegate { toAgentId: B }
                                     ↓
                         Создаётся DelegationRequest (PENDING)
                         Агент B получает WS-уведомление
                                     ↓
              Агент B → PUT /api/delegations/:id/respond { accept: true/false }
                                     ↓
    accept: true                              accept: false
         ↓                                        ↓
  Тикет переназначается             DelegationRequest → REJECTED
  DelegationRequest → ACCEPTED      Агент A получает уведомление
  Новый псевдоним для B
  Пользователь уведомлён
  Агент A уведомлён
```

### Ограничения

- Только текущий `assignee` тикета может инициировать делегирование (ADMIN может для любого тикета)
- Нельзя делегировать самому себе
- Целевой пользователь должен иметь роль `AGENT` или `ADMIN`
- В тикете может быть только один активный `PENDING` запрос одновременно
- Повторное делегирование возможно только после ответа на предыдущий запрос

### После принятия

1. `ticket.assigneeId = toAgentId`
2. Создаётся новый псевдоним для нового агента в этом тикете (`AgentAlias`)
3. WebSocket broadcast в комнату тикета:
   - USER получает: `{ type: "agent_changed", data: { newAgent: "Снежный Пингвин" } }`
   - AGENT/ADMIN получают: `{ type: "agent_changed", data: { newAgent: "Петров П.П.", newAgentAlias: "Снежный Пингвин" } }`
4. Персональное WS-уведомление пользователю: "Теперь вашим обращением занимается Снежный Пингвин"
5. Персональное WS-уведомление бывшему агенту: "Делегирование принято"

## ML-группировка тикетов

### Назначение

Автоматическая классификация тикетов по тематике без ручной разметки. Помогает агентам находить похожие обращения и отслеживать паттерны проблем.

### Модель

- **Библиотека:** `@xenova/transformers` (запускается в Node.js без Python)
- **Модель:** `Xenova/all-MiniLM-L6-v2` (портированная из HuggingFace)
- **Загрузка:** ~80 МБ, скачивается при первом обращении, кэшируется локально
- **Векторное представление:** 384-мерный float-вектор, нормализованный (pooling: mean)

### Алгоритм классификации нового тикета

1. Конкатенация: `text = title + " " + description`
2. `embedding = generateEmbedding(text)` — 384-мерный вектор
3. Загрузка всех `TopicGroup` из БД
4. Для каждой группы вычисляется `cosineSimilarity(embedding, group.embedding)`
5. Если `maxScore >= порог (0.75)`:
   - Тикет привязывается к этой группе (`topicGroupId`)
   - Embedding группы обновляется скользящим средним: `newEmb[i] = (oldEmb[i] * count + emb[i]) / (count + 1)`
   - `ticketCount++`
6. Если `maxScore < порог`:
   - Создаётся новая группа (`autoGenerated: true`)
   - Имя группы: первые 5 значимых слов заголовка (без стоп-слов)
   - Embedding группы = embedding тикета

### Фоновая классификация

После `POST /api/tickets` HTTP-ответ возвращается немедленно. Классификация запускается асинхронно. Если ML-модель недоступна (ещё загружается или ошибка), тикет создаётся без группы.

### Переклассификация

`POST /api/topic-groups/reclassify` (только ADMIN):
1. Все `autoGenerated: true` группы удаляются
2. У всех тикетов `topicGroupId = null`
3. Тикеты обрабатываются последовательно в порядке создания
4. Прогресс логируется в консоль сервера

### Настройка порога

Порог сходства управляется через `SystemSettings` (`grouping.similarityThreshold`, по умолчанию `0.75`).

- Ближе к `1.0` — меньше групп, более строгое объединение
- Ближе к `0.0` — больше групп, более свободное объединение

### Управление группами (ADMIN)

| Действие | Описание |
|----------|----------|
| Переименовать | Задать понятное имя вместо авто-сгенерированного |
| Удалить | Тикеты группы теряют привязку |
| Объединить | Тикеты из группы B переходят в группу A, B удаляется |
| Переклассифицировать | Полный сброс авто-групп и повторная классификация |

## Push-уведомления

### 3 канала доставки

| Канал | Когда работает | Хранение |
|-------|---------------|----------|
| База данных | Всегда | Таблица `notifications` |
| WebSocket | Когда пользователь онлайн | В памяти (доставка в реальном времени) |
| Web Push (VAPID) | Когда подписан + браузер поддерживает | Подписки в `push_subscriptions` |

Каждое уведомление сначала сохраняется в БД, затем отправляется по WS, затем по Web Push (если подписан). Это гарантирует, что уведомление не потеряется, даже если пользователь был офлайн.

### 7 типов уведомлений

| Тип | Получатель | Когда |
|-----|-----------|-------|
| `NEW_MESSAGE` | Пользователь | Агент написал в чат |
| `NEW_MESSAGE` | Агент | Пользователь написал в чат |
| `STATUS_CHANGED` | Пользователь | Статус тикета изменён |
| `TICKET_ASSIGNED` | Агент | Тикет назначен на него |
| `DELEGATION_REQUEST` | Агент | Ему предлагают тикет |
| `DELEGATION_ACCEPTED` | Агент | Его запрос принят |
| `DELEGATION_REJECTED` | Агент | Его запрос отклонён |
| `AGENT_CHANGED` | Пользователь | После принятого делегирования |

### VAPID (Web Push)

VAPID-ключи генерируются автоматически при первом запуске после Setup и хранятся в `SystemSettings` (`push.vapidPublicKey`, `push.vapidPrivateKey`). Ключи не требуют перегенерации при перезапуске сервера.

Протухшие подписки (HTTP 410/404 от браузерного Push-сервиса) автоматически удаляются из `push_subscriptions`.

### Управление подписками (фронтенд)

1. `GET /api/push/vapid-key` — получить публичный ключ
2. Регистрация Service Worker и подписка через `PushManager.subscribe`
3. `POST /api/push/subscribe` — сохранить подписку на сервере
4. Отписка через `DELETE /api/push/subscribe`

## Шифрование вложений

### Назначение

Файлы хранятся в зашифрованном виде. Даже при получении прямого доступа к файловой системе сервера прочитать содержимое файлов без мастер-ключа невозможно.

### Алгоритм: AES-256-GCM

AES-256-GCM — аутентифицированное шифрование. Помимо конфиденциальности обеспечивает целостность: при подмене зашифрованных данных расшифровка завершится ошибкой.

### Параметры

| Параметр | Значение |
|----------|----------|
| Алгоритм | `aes-256-gcm` |
| Длина ключа | 32 байта (256 бит) |
| Длина IV | 16 байт (случайный для каждого файла) |
| Auth Tag | 16 байт |

### Структура файла на диске

```
Байты 0-15:    AuthTag (16 байт)
Байты 16+:     Зашифрованные данные
```

### Именование файлов

```
Загруженный файл:  "отчёт за март 2026.xlsx"
На диске:          "3a7f2c1d-5e89-4b12-a34f-6c890d1e2f34"  (UUID, без расширения)
```

### Хранение метаданных (таблица `Attachment`)

| Поле | Содержимое |
|------|-----------|
| `originalName` | Оригинальное имя файла |
| `mimeType` | MIME-тип |
| `size` | Размер в байтах |
| `storedName` | UUID файла на диске |
| `encryptionIV` | IV в hex-кодировке (32 символа) |

### Мастер-ключ

Генерируется один раз в процессе Setup (шаг 3 — Storage). Хранится в `SystemSettings` как hex-строка (`storage.encryptionKey`). При утере ключа все зашифрованные файлы становятся недоступными.

## Поддержка PostgreSQL и SQLite

### PostgreSQL

Рекомендуется для production-инсталляций с высокой нагрузкой. Поддерживает:
- Конкурентные запросы
- `ILIKE` для регистронезависимого поиска
- Эффективные индексы
- Хранение JSONB (embeddings, push-ключи)

### SQLite

Рекомендуется для небольших команд или single-user инсталляций. Преимущества:
- Не требует отдельного сервера БД
- Файл базы данных — один файл на диске
- Подходит для разработки и тестирования

Ограничения SQLite:
- Нет `ILIKE` — используется `LIKE` (не чувствителен к регистру для ASCII, но не для кириллицы)
- Ограниченная конкурентность при записи

Выбор СУБД производится один раз при прохождении Setup Wizard и записывается в `backend/.env`.

## Автоматическая смена статуса

При отправке сообщения в WebSocket-чат происходит автоматическое изменение статуса тикета:

| Условие | Действие |
|---------|----------|
| USER написал, статус = `WAITING_FOR_USER` | Статус → `IN_PROGRESS` |

Это позволяет агенту видеть, что пользователь ответил на его запрос, без ручного отслеживания.

## Система тикетов

### Нумерация

Каждый тикет получает целочисленный номер (`number`) через hook `beforeCreate`. Номер автоинкрементируется: `MAX(number) + 1`. Номер уникален и не изменяется — используется в интерфейсе для ссылок на тикеты (`#42`).

### Статусы тикетов

```
OPEN → IN_PROGRESS → WAITING_FOR_USER → IN_PROGRESS → RESOLVED → CLOSED
         ↑_______________________________|
```

| Статус | Описание |
|--------|----------|
| `OPEN` | Новое обращение, агент не назначен |
| `IN_PROGRESS` | Взят в работу |
| `WAITING_FOR_USER` | Агент ждёт ответа от пользователя |
| `RESOLVED` | Проблема решена |
| `CLOSED` | Закрыто (финальный статус) |

### Приоритеты

`LOW` < `MEDIUM` < `HIGH` < `CRITICAL`

По умолчанию при создании: `MEDIUM`.
